# Page Specs

Use this page matrix as the Coalition default for ecommerce documentation. Start here, then adjust based on the actual repo and storefront features.

## General

Use for project-wide context and editor onboarding.

Usually include:

- project or theme name
- developer or environment notes
- page inventory
- Page Builder or CMS overview
- design vs preview/publish workflow
- custom widgets overview

Include whenever the project needs a central onboarding page.

## Home

Use for homepage-specific sections and widgets.

Usually include:

- hero banner or carousel
- featured products
- banners and promotional sections
- logo lists or trust sections
- homepage-only custom widgets

Add one `##` section per editable homepage area.

## Category

Use when category or collection pages have meaningful client-managed behavior.

Usually include:

- title and description
- filters and sort behavior
- product listing area
- merchandising or collection-specific custom elements

Omit only if the project has no custom category guidance beyond standard platform behavior.

## Product

Use when PDP content structure or merchandising rules need explanation.

Usually include:

- short description
- gallery/media notes
- product tabs or accordions
- related items
- shipping, availability, or specification content rules
- any required description formatting conventions

## Static

Use for reusable content pages, landing pages, and non-commerce informational pages.

Usually include:

- which static pages are editable
- whether they are built with native page content or Page Builder
- any reusable content blocks or widget rules

Keep it concise if the project has only basic static pages.

## Mega Menu

Use when navigation is custom or visually complex.

Usually include:

- how the menu content is managed
- category/image/banner relationships
- any custom widget or menu-specific content source

Omit if the theme uses only a stock navigation with no extra instructions.

## Other

Use as a catch-all page for project-specific features that do not fit the main storefront pages.

Examples:

- search
- cart or mini-cart
- footer
- blog
- custom badges
- global promo bars

Prefer adding a new dedicated page instead of overloading `Other` when the topic is large enough.

## Add or omit logic

- Add a page when the repo contains a distinct editable feature area that would otherwise make another page too broad.
- Omit a page when there is no custom guidance and no likely user need.
- Merge small topics into `Other` only when they do not justify their own page.
- If the repo already has a nav, preserve that wording unless the user asks for a cleanup.
